技術深挖 (Technical Deep Dive)

1. Webview (Client) 與遠端 Extension (Host) 檔案系統隔離與傳輸方案

在 VS Code Remote-SSH 開發環境中，Webview 執行於本地學生的電腦（Client 端），而 Extension 的 Node.js 進程則運行於遠端伺服器（Host 端）。此架構下，Extension 的 fs 模組操作的完全是遠端主機的磁碟，無法直接讀取學生的本地影像。

為了解決此隔離屏障，我們採用了 FileReader (ArrayBuffer) + Base64 封裝。由 Webview 本地端藉由 <input type="file"> 讀取檔案，並利用分塊防溢出循環：

let base64 = '';
const chunk = 8192;
for (let i = 0; i < bytes.length; i += chunk) {
    base64 += String.fromCharCode.apply(null, bytes.subarray(i, i + chunk));
}
base64 = btoa(base64);

將二進位資料轉換為安全的 Base64 字串後，透過 VS Code Webview 的 postMessage 管道跨越網絡安全地發送至遠端 Extension。此做法既安全、輕量，又完全避開了需引入複雜的 SFTP 或 SSH 連線套件。

2. 遠端原位解壓縮 (Remote In-place Decryption/Extraction) 設計

將 ZIP 上傳至遠端後，如何解壓縮是另一個關鍵問題。如果依賴遠端 Linux 上的 unzip 指令，會面臨某些極簡 Docker 容器或伺服器環境未安裝該軟體的情況；若在 Extension 引入 node.js 的第三方 zip套件，又會讓 VSIX 套件體積顯著膨脹。

我們的解決方案是呼叫遠端 Python 原生的 zipfile 模組。由於專案為 AI 學習平台，遠端主機必然已安裝 Python 環境。我們透過 Node.js 的 execFile 執行以下一鍵 Python 腳本：

import zipfile
import os
try:
    with zipfile.ZipFile("temp.zip", 'r') as zip_ref:
        zip_ref.extractall("target_dir")
    os.remove("temp.zip")
    print("SUCCESS")
except Exception as e:
    print("ERROR:", str(e))

此設計確保了解壓縮機制具有極高的跨平台相容性與零依賴特性，完美達成「寫入遠端時直接解壓且維持 Extension 輕量化」之預期目標。

3. VS Code 遠端 Server 環境下 PATH 變數缺失與 exec 執行權限探討

在 VS Code Remote-SSH 連線時，其所啟動的 vscode-server 常採用非交互非登入式 (Non-interactive non-login) Shell 來掛載 Extension 進程。在這種 Shell 中，系統不會完整載入 /etc/profile 或使用者的 .bashrc，導致 Extension Node.js 的 process.env.PATH 缺乏諸多常用的軟體二進位目錄（如 /usr/sbin, /usr/local/bin 或 /snap/bin）。

此問題導致即便使用者手動在 SSH 終端機能夠成功呼叫 nvidia-smi，Extension 執行 exec('nvidia-smi') 時依然會出現 command not found 的異常。我們在 executeCommand 中實作了環境變數注入：

const env = Object.assign({}, process.env);
if (process.platform !== 'win32') {
    const extraPaths = '/usr/bin:/usr/sbin:/usr/local/bin:/usr/local/cuda/bin:/snap/bin';
    env.PATH = env.PATH ? `${env.PATH}:${extraPaths}` : extraPaths;
}
exec(cmd, { env }, ...)

藉由將這些關鍵路徑強制追加至執行的環境變數中，成功讓遠端的 nvidia-smi 及 docker 等指令被精準定位並順利執行，保障了診斷功能的強健度。
