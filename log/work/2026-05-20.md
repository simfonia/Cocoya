# Cocoya 工作日誌 (2026-05-20)

## 今日進度
### 1. [Bug 修復] 積木欄位焦點與同步問題
- **問題描述**：使用者在積木欄位輸入值後，若未按下 Enter 或使欄位失去焦點便直接點擊「執行」或「上傳」，Blockly 尚未將新值寫入積木屬性，導致產生的程式碼仍為舊值。
- **修復方案**：
    - **[UI 層]**：在 `ui/src/ui/base.js` 的工具列按鈕點擊事件 (`bind` 輔助函式) 開頭加入 `Blockly.hideChaff()`。這會強制關閉所有開啟中的 Widget (如文字輸入框)，觸發 Blockly 的值寫回機制。
    - **[App 層]**：在 `ui/src/app/workspace.js` 中新增 `triggerCodeUpdateSync()` 方法。
        - 理由：原本的 `triggerCodeUpdate` 有 300ms 的 debounce 延遲。若在執行時僅依賴 `lastCleanCode`，可能會拿到尚未更新的舊代碼。
        - 實作：同步執行 `workspaceToCode` 與正則清理邏輯，確保「執行」動作能立即獲取最新代碼。
- **效果**：點擊任何工具列按鈕時，輸入框會立即收合且新值保證生效，徹底解決使用者誤會。

### 2. [功能開發] 支援 Geek 270 度舵機與安全限位機制
- **需求**：πCar 需要能支援 Geek 270 舵機，但必須防止在小車機構上轉動超過 180 度導致干涉損壞；同時在非小車機構上需能使用全行程。
- **實作內容**：
    - **[Blockly]**：重構 `mcu_car_servo_setup` 積木。新增「型號選單」，提供 **標準 (180°)**、**Geek 270 (全行程)** 與 **Geek 270 (限 180°)** 三種預設值，並透過 `validator` 實現數值自動填入。
    - **[i18n]**：更新語系檔，移除通用舵機積木上的 `(0-180)` 硬編碼提示，改為由設定決定。
    - **[Generator]**：升級 `PiCarServo` Python 類別。
        - 支援 `max_angle` 動態配置與 `_CONFIG` 全域映射。
        - **安全限位 (Safety Valve)**：在 `set_angle` 與 `move_sync` 中加入 `is_car_action` 旗標。
        - 針對「雙手動作」等小車專屬積木，強制將輸出限制在 0-180 度，確保物理安全。
        - 通用設定角度積木則解鎖全行程限制，增加使用彈性。

### 3. [環境優化] VS Code 偵錯配置重構
- **問題**：F5 啟動偵錯時偶爾出現 "Canceled" 對話框，主因為預設的 `npm: compile` 任務與 VS Code 自動偵測名稱衝突。
- **改動內容**：
    - 將 `.vscode/tasks.json` 的任務標籤改為明確的 `compile-extension`。
    - 將執行型態由 `npm` 改為 `shell` 以提升命令執行的穩定性。
    - 同步更新 `.vscode/launch.json` 的 `preLaunchTask` 配置。
- **效果**：消除 F5 啟動時的競賽條件與名稱衝突，確保編譯流程強健。

### 4. [範例更新] 同步更新 270 度舵機相關範例
- **內容**：針對 `examples/` 目錄下的範例 35、36、36-1 與 37 進行 XML 結構更新。
- **改動**：
    - 在所有舵機積木中補上 `<field name="TYPE">` 欄位。
