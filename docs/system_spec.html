Cocoya系統規格說明書 v3.4

Code, Compute, Yield AI - 專為 AI 教育設計的視覺化開發環境。

1. 專案概述

Cocoya 是一個整合了 Blockly 與 Python (OpenCV/MediaPipe) 的 VS Code 擴充功能。它具備跨平台支援能力，可開發 PC 端 AI 程式或 MicroPython 硬體程式。

核心技術

  • Extension Host: TypeScript 4.9+ (VS Code API)
  • Frontend: Vanilla JavaScript (ES6)
  • Logic Engine: Google Blockly v12.x
  • Parallel Loading: Promise.all 模組載入機制

2. 開發規範

3. 檔案架構 (v6.0 模組化 SSOT 版)

為了解耦龐大的編輯器邏輯，系統將原本的單一 JS 檔案拆分為功能專一的子模組。所有代碼存放於 ui/src/ 並遵循 SSOT 原則。

cocoya/ui/src/
├── bridge/ (通訊橋樑)
│ ├── base.js, tauri.js, vscode.js
├── app/ (核心邏輯)
│ ├── persistence.js (存檔/備份)
│ ├── workspace.js (積木引擎)
│ └── lifecycle.js (初始化)
├── ui/ (介面元件)
│ ├── terminal.js, renderer.js
│ └── dialogs.js (自定義彈窗)
├── modules/ (雙模共用模組)
├── zh-hant.js (核心語系)
└── style.css (全域樣式)

13. 雙模架構開發規範 (Dual-Mode Guidelines)

Cocoya 具備 VSIX 與 Tauri 雙重引擎。為防止兩者在開發過程中互相干涉或功能脫節，必須嚴格遵守以下準則：

A. 後端功能對齊 (Feature Parity)

開發任何涉及「系統層級」的功能（如：檔案存取、代碼執行、硬體通訊）時，必須確保 VS Code (TypeScript) 與 Tauri (Rust) 兩端皆有對等的指令實作。

  • 規範: 前端一律透過 CocoyaBridge 發送指令，禁止在 UI 邏輯中直接撰寫環境專屬 API。
  • 連動性: 修改完 extension.ts 的 Command 後，應立即在 src-tauri/src/lib.rs 補齊對應的 Rust 指令。

B. 檔案路徑與權限處理

VS Code 受限於 Webview 沙盒與插件權限，而 Tauri 具備完整系統權限。為確保相容性：

  • 路徑規範: 儘量避免使用絕對路徑。若需儲存暫存檔，應優先使用系統提供的 temp 或 appData 目錄。
  • 校驗機制: 在 Tauri 模式下成功的路徑操作，必須同步在 VS Code 模式下進行回歸測試。

C. 編譯產物與隔離 (.vscodeignore)

為了防止不同模式的編譯廢料互相干涉，系統實作了嚴格的排除機制：

  • VSIX 打包: 透過 .vscodeignore 強制排除 src-tauri/ 與 node_modules/，維持插件輕量化 (~1MB)。
  • Tauri 打包: Rust 編譯產物 (target/) 嚴禁進入 VSIX 封裝；VSIX 的 TypeScript 編譯產物 (out/) 亦不應影響 Tauri 運行。

4. 模組化與平行載入

系統採用 Promise.all 平行載入所有模組。順序由 core_manifest.json 決定，確保 Toolbox 分類順序一致且載入效能最優化。

ID 命名慣例

檔案命名必須遵循 [pure_id]_blocks.js 與 [pure_id]_generators.js。載入器會自動根據 ID 分段 (split) 並提取最後一部分作為檔名。

5. 語系分散化管理 (i18n)

為了解決單一語系檔過大的問題，採用分散式儲存：

  • 全域語系: 位於 media/zh-hant.js，定義 UI 按鈕、對話框與分類總稱。
  • 模組語系: 位於 modules/[moduleId]/i18n/[lang].js，僅定義該模組特有的積木文字。

載入器會在載入積木前，優先嘗試載入對應的模組語系檔。

佔位符與 Blockly 解析規範 (Placeholder Resolution)

Blockly 解析 %{BKY_XXX} 佔位符時，會自動去掉 BKY_ 前綴，然後查詢 Blockly.Msg["XXX"]。因此：

  • toolbox.xml: 必須使用 %{BKY_XXX} 格式（有 BKY_ 前綴）。
  • Blockly.Msg 定義: 必須定義 Blockly.Msg["XXX"]（無 BKY_ 前綴）。

範例：

// 正確定義
Blockly.Msg["CAT_AI_INFERENCE"] = "AI 推論";
Blockly.Msg["COLOUR_AI_INFERENCE"] = "#9C27B0";

這樣 Blockly 解析 %{BKY_CAT_AI_INFERENCE} → 查詢 Blockly.Msg["CAT_AI_INFERENCE"] → 正常顯示「AI 推論」。

6. 平台模式與自動偵測

Cocoya 支援 PC (Python) 與 MicroPython (MCU) 雙模式隔離。

模式資訊會自動注入 XML 屬性中，實現「開檔即鎖定模式」的功能。

積木級別平台過濾 (Platform Filtering)

透過在 toolbox.xml 的 <block> 標籤中加入 platform="PC" 或 platform="MCU" 屬性，系統在切換平台時會自動過濾並重新載入 Toolbox，確保當前模式下僅顯示適配的積木。

7. 效能優化 (State Check)

積木啟用狀態 (孤兒檢查) 採用 Top-Down 遞迴演算法。運算量與頂層積木數成線性關係 ($O(N)$)，顯著改善大規模專案下的拖拽流暢度與垃圾桶動畫效能。

8. 硬體部署系統 (MCU)

部署邏輯封裝於 resources/deploy_mcu.py 中，支援透過磁碟 (CIRCUITPY) 或 Serial REPL 進行傳輸，並整合即時 Monitor。

9. 積木定位技術 (Regex Deep Dive)

為了達成「積木」與「產出代碼」之間的精確導航，Cocoya 採用了區塊範圍 ID 注入機制。這套機制保證了積木與代碼區塊之間的 1:1 對齊，支援整段高亮功能。

A. 全域注入機制 (Defensive scrub_)

定位標記由 media/main.js 重寫 Blockly.Python.scrub_ 函式。為了防止連線屬性缺失導致的崩潰，系統實作了嚴格的連接點檢查。

// main.js 中的核心邏輯 (v2.6 防禦性遍歷版)
Blockly.Python.scrub_ = function(block, code, opt_thisOnly) {
    const nextBlock = (block.nextConnection && !opt_thisOnly) ? block.nextConnection.targetBlock() : null;

    // 1. 處理不啟用的積木：遞迴跳過並抓取下一個
    if (!block.isEnabled()) {
        return nextBlock ? Blockly.Python.blockToCode(nextBlock) : '';
    }
    
    const s = window.CocoyaUtils.TAG_START;
    const e = window.CocoyaUtils.TAG_END;

    // 2. 區分積木類型以注入標記
    if (block.outputConnection) {
        // 運算式 (Expression): 絕不尋找 nextBlock，防止 TypeError
        return `${s}ID:${block.id}${e}${code}`;
    } else {
        // 陳述句 (Statement): 連接下一個積木並注入範圍標記
        const nextCode = nextBlock ? Blockly.Python.blockToCode(nextBlock) : '';
        let lines = code.split('\n');
        if (lines.length > 0 && lines[lines.length - 1] === '') lines.pop();
        if (lines.length > 0) {
            lines[0] += `  # S_ID:${block.id}`;
            lines[lines.length - 1] += `  # E_ID:${block.id}`;
        }
        return lines.join('\n') + '\n' + nextCode;
    }
};

B. 標記類型與正則表達式

標記類型正則表達式用途
範圍開始/ # S_ID:([^\s\n]+)/g標記積木產出代碼的首行。
範圍結束/ # E_ID:([^\s\n]+)/g標記積木產出代碼的末行。
運算式/單行/ # ID:([^\s\n]+)/g標記無子區塊的單行積木或行內運算。

C. 處理流程 (The Workflow)

UI 渲染器處理流程：

  1. 解析: 掃描每一行，若發現 S_ID 則記錄該積木的起始行號，發現 E_ID 則更新結束行號。
  2. 映射: 建立 blockId -> {start, end} 映射表。
  3. 渲染: 當使用者點擊積木時，UI 管理器會根據映射表循環高亮該範圍內的所有 DOM 行。
  4. 抹除: 在執行代碼或複製到剪貼簿前，系統會使用正則徹底清除所有標記。

10. AI 視覺模組規範 (MediaPipe)

為了確保 MediaPipe 與 OpenCV 的整合穩定性，AI 模組須遵循以下實作標準：

  • 物件導向初始化: 產生器必須完整產出初始化代碼（包含 import、mp.solutions 初始化），以便使用者在全域定義區直接使用。
  • 座標歸一化 (Normalization):
    • 原始資料傳回 MediaPipe 座標 (0.0 - 1.0)。
    • 提供專用的 get_landmark_xy 積木進行像素轉換。
  • 強型別連線檢查 (Type Safety):
    • 偵測結果使用 HandLandmarks 與 Landmark 自定義類型。
    • 確保「取得特徵點」積木與「座標轉換」積木之間的連線安全性。

11. AI 推論與訓練模組規範 (Template-Based Architecture)

為了實現模組化、可維護且易於擴展的 AI 訓練系統，Cocoya 採用基於外部 Python 模板的架構設計。

A. 模板化訓練系統 (Template-Based Training)

訓練邏輯完全外部化到 resources/train_templates/ 目錄，每個任務類型獨立維護：

resources/train_templates/
├── classifier/
│ └── classifier_train.py
├── detector/
│ └── detector_train.py
└── line_follower/
└── line_follower_train.py

B. 統一介面規範 (Unified Interface)

所有訓練模板必須遵循統一的命令列介面：

python classifier_train.py \
    --dataset_dir  \
    --output_dir  \
    --epochs  \
    --batch_size  \
    --learning_rate  \
    --project_name

C. 產生器路由邏輯 (Generator Routing)

JavaScript 產生器僅負責參數驗證與模板呼叫：

// 根據任務類型選擇模板
if (task_type == "classifier") {
    script_name = "classifier_train.py";
} else if (task_type == "detector") {
    script_name = "detector_train.py";
}

// 呼叫 subprocess 執行
cmd = [
    sys.executable,
    script_path,
    "--dataset_dir", dataset,
    "--output_dir", output_dir,
    "--epochs", str(config["epochs"]),
    ...
];
result = subprocess.run(cmd, capture_output=True, text=True, shell=True);

D. 跨平台兼容性 (Cross-Platform Compatibility)

使用 shell=True 參數確保在 Windows、Git Bash、macOS 和 Linux 下均能正常執行：

# 支援 Windows & Unix-like 系統
result = subprocess.run(cmd, capture_output=True, text=True, shell=True)

E. 結果回傳格式 (Result Format)

訓練模板必須在執行結束時輸出 JSON 格式的結果：

RESULT: {
    "success": true,
    "projectName": "gesture_classifier",
    "modelDir": "model/",
    "epochs": 50,
    "batchSize": 32,
    "learningRate": 0.001,
    "accuracy": 0.935
}

F. 模板開發規範 (Template Standards)

  • 獨立性: 每個模板必須能獨立執行，不依賴 Cocoya 環境
  • 參數驗證: 必須包含完整的參數驗證與錯誤訊息
  • 進度回饋: 提供詳細的訓練進度與結果輸出
  • 模型輸出: 必須產出標準格式的 TFLite 模型與 labels.txt

G. 版本控制與相容性 (Versioning)

模板版本號記錄在 resources/train_templates/version.json：

{
    "classifier": "1.0.0",
    "detector": "0.1.0",
    "line_follower": "0.0.1"
}

11. 縮排與代碼產生規範

為了解決不同開發環境下的縮排位移問題，系統建立了嚴格的產生器標準：

A. 動態縮排 (Dynamic Indentation)

嚴禁在產生器中寫死空格（如 ' pass\n'）。必須使用 generator.INDENT 以對齊 UI 設定。

// 正確做法
    var branch = generator.statementToCode(block, 'DO') || generator.INDENT + 'pass\n';

B. 基準縮排與自動縮放 (Base 4-Space & Scaler)

強制規範: 所有在產生器 (.js) 中以靜態字串形式編寫的 Python 程式碼，必須統一使用 4 個空白 (4 spaces) 作為基準縮排。

系統在 ui/src/utils.js 的 finish 階段實作了全域縮排縮放器。它會掃描所有 definitions_，將基準 4 空白按比例縮放為目前的縮排長度。這確保了無論縮排設定為何，全域區的程式碼都能正確對齊。

C. 全域定義區行為 (Definition Zone)

py_definition_zone 積木具備自動移除縮排機制。它會移除內部積木產出的所有行首縮排，使 import 與全域初始化代碼能正確靠左對齊。

13. 非同步對話框機制 (Bridge)

由於 VS Code Webview 禁止使用同步的 window.prompt(), confirm() 與 alert()，系統實作了一套跨端非同步橋樑：

  • 核心技術: 透過 Blockly.dialog.setPrompt() 與 setConfirm() 攔截核心事件。
  • 通訊流程:
    1. Webview 發送 postMessage 至 Host。
    2. Host 呼叫 vscode.window.showInputBox 或 showInformationMessage。
    3. 使用者輸入後，Host 回傳結果並帶上唯一的 requestId。
    4. Webview 透過 Map 儲存的回調函式 (Callback) 異步完成操作。
  • 規範: 禁止在任何積木產生器或 UI 邏輯中使用同步彈窗，必須遵循此非同步回調規範。

13. 變數管理與動態回調

為了解決變數建立與重新命名的同步性問題：

  • 動態 Toolbox: 變數分類採用 registerToolboxCategoryCallback 動態產生，允許注入「建立變數」等自定義按鈕。
  • 更名邏輯: 透過上述的非同步橋樑，系統能正確處理 Blockly 核心的「重新命名變數」選單，實現無感知的 Host-Webview 同步。

14. 產生器引號規範 (Quote Standards)

為了避免使用者在積木中輸入引號與產生器外層引號衝突（導致 Python SyntaxError），系統建立以下規範：

A. 靜態字串與變數

一般 text 積木或簡單變數賦值，統一遵循 Blockly 預設之單引號 '...' 規範。底層透過 Blockly.Python.quote_ 處理內容轉義。

B. 複雜組合與 f-string (The Silver Bullet)

針對 py_text_join 等涉及多個子積木拼接的運算式，產生器必須使用三雙引號 f"""..."""。這能保證內部即便包含字典鍵值（如 ['key']）或使用者手打的雙引號，皆不會引發語法錯誤。

C. 規範範例

# 正確的產生結果
print(f"""Alice 的分數是: {scores_dict['Alice']}""")

15. 環境診斷與套件管理

為了解決 AI 教育中常見的 Python 環境設定挫折，系統內建了自動化診斷與安裝助手。

A. 偵測機制 (Diagnostic Logic)

當使用者點擊診斷按鈕時，Webview 會發送 checkEnvironment 訊息至 Host 端。Host 端會針對 cv2, mediapipe, PIL, serial 執行背景導入測試：

// Host 端的模組探測實作
execSync(`"${pythonPath}" -c "import ${module}"`, { stdio: 'ignore' });

B. 一鍵安裝 (One-click Installation)

針對偵測為「未安裝」的模組，系統會呼叫 -m pip install 命令，並自動開啟 VS Code 整合終端機 Cocoya Environment 顯示進度，安裝後自動觸發重新掃描。

16. 效能優化 (圖片快取)

針對 Overlay Image 積木在高頻 AI 迴圈中可能導致的 FPS 低落問題，系統實作了全域圖片快取機制。

A. 緩存 Key 產生 (Cache Matching)

系統根據 Path + Width + Angle 生成唯一雜湊值。只有當參數變動或首次載入時，才會執行耗時的磁碟讀取 (imread) 與仿射變換 (warpAffine)。

B. 記憶體管理 (GC Policy)

產生器會在產出的代碼中自動維護 cocoya_img_cache 字典。當快取項次超過 20 個時會執行自動排空，防止長時間執行導致的記憶體溢出。

17. 積木視覺外觀雙軌制 (Visual Style)

為了解決「學習程式碼」與「快速上手應用」之間的平衡，Cocoya 採用雙軌制外觀規範：

A. 核心語法模組 (Source Code Style)

  • 適用模組: structure, io, logic, loops, math, text, types, variables, functions。
  • 設計原則: 盡量維持 Python 原始碼關鍵字與標點符號。例如 import, if, for, print()。
  • 目的: 建立學生對真實 Python 代碼的肌肉記憶與視覺聯繫。

B. 應用與硬體模組 (Friendly UI Style)

  • 適用模組: mcu_car, ai_hand, cv_draw, hardware 等。
  • 設計原則: 以自然語言（繁中/英文）描述功能。例如「控制小車 [前進] 速度 [50]」而非 car.move(1, 50)。
  • 目的: 降低特定硬體或複雜 AI 功能的操作門檻，讓學生專注於邏輯設計。

18. 自定義積木搜尋引擎 (BlockSearcher)

為了解決積木數量過多導致的尋找困難，系統實作了專為 Webview 優化的模糊搜尋引擎。

A. 模糊文字索引機制 (Fuzzy Indexing)

  • 索引建立: 系統在啟動時會遍歷所有已載入模組，透過 workspace.newBlock(type) 建立臨時積木，並提取所有 Field 中的真實顯示文字（包含 i18n 翻譯內容）。
  • 非破壞性探測: 索引建立期間強制呼叫 Blockly.Events.disable()，確保地圖或其他監聽插件不會因頻繁建立/刪除積木而報錯。

B. 嵌入式 UI 注入技術

搜尋框透過 MutationObserver 監控 Toolbox 生成，並動態將 HTML 搜尋欄位插入至側邊欄清單頂端，實現與 Toolbox 的一體化滾動與視覺對齊。

19. πCar 高精度控制與並行 (Precision Control)

針對 Maker Pi RP2040 硬體平台，系統實作了工業級的驅動邏輯與音樂引擎。

A. 馬達與舵機精準化

  • 馬達邏輯: 實測校正邏輯為 Low=Brake (煞車), High=Coast (滑行)，確保小車停止時動作乾脆。
  • 同步舵機移動: 實作 move_sync 並行演算法，透過單一 while 迴圈與微步進，解決左右手依序動作的延遲感，實現雙手同步合攏/張開。

B. 旋律解析器 (Regex Melody Parser)

在 Python 注入 MusicEngine 類別，使用高效正則表達式解析字串旋律（如 C4Q, D4H.），支援八度、時值、附點與三連音，大幅降低音樂創作門檻。

20. 全局 SVG 屬性安全攔截 (NaN Protection)

為了解決複雜插件（如 Minimap）在工作區渲染未完成時計算錯誤導致的 Webview 崩潰問題，系統實作了底層攔截器。

A. 攔截原理

在 main.js 入口處徹底覆寫 Element.prototype.setAttribute，攔截所有包含 NaN 或 Infinity 的字串或數值輸入。這從底層保障了不論哪種第三方積木或插件出錯，皆不會導致瀏覽器渲染異常或報錯刷屏。

21. 積木說明文件系統 (Block Help System)

為了解決硬體腳位配置複雜導致的學習門檻，系統建立了整合式的圖形化說明機制。

A. 本地路徑解析橋樑

由於 Webview 安全限制，系統在 main.js 攔截 window.open 請求，將相對路徑的 Help ID 傳送至 Host 端。Host 端透過 vscode.env.openExternal 與 file:// 協定開啟 media/docs/ 下的 HTML，確保圖片與超連結在外部瀏覽器中正常運作。

B. 多語系自動匹配

說明文件採用 [id]_[lang].html 命名規範（如 hardware_pins_zh-hant.html）。攔截器會根據當前 Webview 語言自動附加語系後綴，提供在地化的學習資源。

22. 雙模開發架構 (Dual-Mode Architecture)

為了兼顧開發者環境與終端教學現場，Cocoya 採用一套 UI、兩套 Host 的雙模架構。

A. 模組化累積機制 (Object.assign Pattern)

前端採用 累加式對象定義。各子模組透過 window.CocoyaApp = Object.assign(window.CocoyaApp || {}, { ... }) 將功能注入全域命名空間。

  • 優點: 無需現代 Bundler (如 Vite/Webpack) 即可在瀏覽器環境直接執行。
  • 安全性: 徹底解決多檔案載入時發生的命名空間覆蓋 (Overwrite) 問題。

B. 通訊橋樑 (CocoyaBridge)

系統透過 bridge.js 抽象化底層通訊。前端邏輯呼叫 Bridge.saveFile() 時，內部會自動判別環境：

  • VSIX 模式: 呼叫 vscode.postMessage({ command: 'saveFile' })。
  • Tauri 模式: 呼叫 invoke('save_file') (Rust backend)。

C. 功能鏡像 (Feature Mirroring)

Rust 後端會實作與 VS Code Extension Host 完全對等的功能（如：序列埠掃描、Python 代碼執行、磁碟部署），確保教學體驗的一致性。

23. 多視窗完整性協議 (Multi-Window Integrity Protocol, MWIP)

為了解決多個編輯視窗同時操作同一份資料時的潛在衝突與數據遺失，系統在 Tauri 桌面模式下實作了嚴格的完整性機制：

A. 單一事實來源鎖定 (SSOT File Locking)

後端透過 HashMap<PathBuf, WindowLabel> 追蹤檔案所有權。第一個開啟該路徑的視窗獲得獨佔寫入權，後續視窗強制以「唯讀模式」載入，且無法透過「另存新檔」覆蓋已鎖定的檔案。

B. 原子化髒狀態同步 (Atomic Dirty Sync)

視窗的「髒狀態」必須與 Rust 後端同步。在執行「儲存並關閉」流程時，系統會先 await 後端狀態更新為乾淨，確保 onCloseRequested 攔截器能讀取到正確的最終狀態，防止 Race Condition 導致的重複對話框。

C. 宣示權備份恢復 (Safe Backup Claiming)

為防止多個新視窗搶奪同一個遺留備份檔，系統採用「先到先得」宣示機制：當視窗偵測到備份後，立即將其重新命名為 .recovering。這保證了備份恢復動作的唯一性與不可重複性。

D. 精準事件路由 (Targeted Unicast)

所有關於「視窗即將關閉」或「備份偵測」的事件，必須使用 emit_to(label, ...) 定向發送。嚴禁使用全域廣播，以確保多視窗環境下，使用者操作的交互提示不會發生跨視窗干擾。
